# GitHub Issue 写作最佳实践

基于与 Vivi 协作 AgoraReads 项目的实战经验总结。

## 核心原则

### 1. 绝不限制技术实现

**反面教材（Issue #4 初稿）：**
```markdown
## 技术考虑
- 知识库格式：Markdown（便于 Git 版本控制）
- 相关性判断：由元子或新角色在分析时输出评分
- 自动追加：通过 `bb roam-write` 或直接 `git commit`
```

**问题：** 规定了具体的技术栈、文件格式、实现方式，限制了实施方的探索空间。

**正确做法：**
```markdown
## 设计约束
- 不规定具体实现方式（文件格式、存储位置、通知渠道等）
- 保持轻量，不增加架构复杂度
- 在现有流程中增加步骤，而不是重构整个系统
```

**核心洞察：** 
- Issue 的职责是说清楚"要什么"，不是"怎么做"
- 技术实现是实施方的专业领域，不应该被需求方限制
- 只在有明确约束时才提（比如"必须兼容现有 API"），而不是提供"建议方案"

### 2. 功能性 vs 架构性 Issue 的区分

| 维度 | 功能性 Issue | 架构性 Issue |
|------|-------------|-------------|
| **性质** | 具体功能的添加/修复 | 系统设计原则的变更 |
| **关注点** | "做对了吗" | "做对了事吗" |
| **验收方** | 开发团队（如 AGProd） | 产品负责人（如 Vivi） |
| **示例** | "为元子视角增加子栏目" | "三层分离架构设计" |

**为什么这个区分重要：**
- 功能性 issue：开发团队有技术专业性验证实现质量
- 架构性 issue：需要全局视角验证是否符合整体愿景

**如何判断：**
- 问自己：这个 issue 是在要求"添加一个具体功能"，还是在改变"系统如何工作的基本原则"？
- 如果是后者，标记为架构性，需要产品负责人验收

### 3. 写清楚"要什么"的三要素

#### 要素 1：背景（为什么需要）

**不够好：**
```markdown
## 背景
当前系统没有这个功能。
```

**好的示例：**
```markdown
## 背景
当前系统把三件本质不同的事情混在了一起：
1. 状态监控 — Vivi 想随时看到 agents 的当前状态
2. 周报 — Vivi 想定期收到"这周发生了什么"的总结
3. 出神（自我进化）— agents 审视自己并决定是否修正信念

**问题：** 出神既承担了 agent 自我进化的功能，又是唯一能看到 agent 状态的时机，还部分承担了周报的职责。
```

**关键：** 说清楚当前的痛点是什么，为什么现状不够好。

#### 要素 2：目标（要达到什么结果）

**不够好：**
```markdown
## 目标
实现一个 dashboard。
```

**好的示例：**
```markdown
## 目标
将三件事解耦，让它们独立运行：

### 1. 状态监控（Dashboard）
**性质：** 只读操作，展示当前状态
**需求：**
- Vivi 可以**任何时候**查看每个 agent 的当前状态
- 不需要等出神，不需要等周报
- 包含信息：信念数量、更新率、最近更新、活跃冲突、健康度
```

**关键：** 说清楚成功是什么样子，用户能做什么，系统应该表现出什么行为。

#### 要素 3：验收标准（如何验证成功）

**不够好：**
```markdown
## 验收标准
- [ ] Dashboard 完成
```

**好的示例：**
```markdown
## 验收标准
- [ ] Dashboard 可以随时查看，不需要等出神
- [ ] Dashboard 在每篇文章分析后自动更新
- [ ] Dashboard 包含：信念数/上限、窗口更新率、全局更新率、最近更新、活跃冲突、健康度
- [ ] 周报每周定期产出，包含 co 的判断和建议
- [ ] 出神由 agents 自主触发（基于定期/事件/阈值/自主判断）
- [ ] 三者独立运行，互不依赖
```

**关键：** 可测试、可验证、具体明确。

### 4. 协作式起草流程

**错误模式：**
```
需求方独自写 Issue → 提交 → 开发团队实现
```

**正确模式：**
```
需求方提出想法
    ↓
需求方 + 助手协作讨论
    ↓
起草 Issue（可能多轮迭代）
    ↓
需求方 review 并确认
    ↓
助手提交到 GitHub
    ↓
开发团队实现
```

**关键洞察：**
- Issue 写作是一个**思考和澄清需求**的过程
- 通过讨论，需求方自己也会更清楚真正想要什么
- 助手的价值在于：帮助结构化表达、识别模糊点、提醒遗漏的部分

## 实战案例

### 案例 1：Issue #4（Agent 核心信念系统）

**初稿问题：**
- 混淆了"建立知识库"和"在分析流程中加入视角"
- 技术实现细节过多

**改进后：**
- 明确了是"在 ag-crew 分析流程中加入人机共生视角"
- 去掉了所有技术实现的限制
- 只保留了设计约束（不规定具体实现方式）

**教训：** 和需求方多轮讨论，直到双方都清楚"真正要什么"。

### 案例 2：Issue #6（三层分离架构）

**成功要素：**
- 基于深度讨论（两轮，包含第一性原理分析）
- 清晰的问题陈述（当前把三件事混在一起）
- 明确的目标（三层独立运行）
- 具体的验收标准（7 条可测试的标准）
- 不限制实现方式

**核心原则体现：**
- 说清楚"出神是 AI 自己的事情"（核心原则）
- 说清楚三层各自的性质和需求（目标）
- 说清楚如何验证成功（验收标准）
- 完全不涉及技术实现细节

## 常见陷阱

### 陷阱 1：把"建议方案"写进 Issue

**错误：**
```markdown
可以用 Python 的 schedule 库实现定时任务...
```

**正确：**
```markdown
需要支持定时触发（每周、每 N 篇文章等）
```

### 陷阱 2：验收标准过于模糊

**错误：**
```markdown
- [ ] 系统运行良好
```

**正确：**
```markdown
- [ ] Dashboard 在每篇文章分析后 30 秒内自动更新
- [ ] 更新失败时有明确的错误日志
```

### 陷阱 3：混淆"功能需求"和"实现方案"

**错误：**
```markdown
需要一个 cron job 每天 8 点运行...
```

**正确：**
```markdown
需要每天早上自动生成周报并通知用户
```

## 检查清单

提交 Issue 前，问自己：

- [ ] 我说清楚了"为什么需要"吗？（背景）
- [ ] 我说清楚了"要达到什么结果"吗？（目标）
- [ ] 我说清楚了"如何验证成功"吗？（验收标准）
- [ ] 我有没有限制技术实现？（如果有，是必要的约束吗？）
- [ ] 这是功能性还是架构性 issue？（决定谁来验收）
- [ ] 验收标准是否具体、可测试？
- [ ] 我有没有把"建议方案"误写成"需求"？

## 总结

**好的 Issue 写作 = 清晰的需求表达 + 不限制实现方式**

- 说清楚"要什么"，不说"怎么做"
- 区分功能性和架构性，决定谁来验收
- 通过协作讨论澄清需求
- 用具体的验收标准确保可测试

**记住：** Issue 不是技术文档，是需求和实施方之间的契约。
